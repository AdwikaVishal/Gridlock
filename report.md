# Gridlock — End-to-end execution report (no code changes)

> Scope: describe how the running project works from request entry → backend flow → Python inference → data persistence → how (and whether) updates reach the frontend. Also lists suspected/non-working parts, failures, and major issues observed from the code.

---

## 1) High-level architecture

### Backend (Node/Express + SQLite + Python bridge)
- **Express API** exposes REST endpoints under `/api/*`.
- Detection processing is triggered by HTTP uploads (image/video).
- Python inference runs via `child_process.spawn()`.
- Results are persisted into **SQLite** (`data/traffic_violations.db` by default).
- Analytics and reports are generated by reading from the same SQLite DB.

### Frontend (React + Vite)
- Uses `axios` with `baseURL: '/api'` for REST calls.
- Detection page uploads an image and then **polls job status** until completion.
- A WebSocket client exists in the codebase, but real-time streaming appears **not wired** to the backend detection pipeline.

## 3) Detection pipeline (where data goes)

### 3.1 Upload request (image)
**Endpoint:** `POST /api/detection/image`

**File:** `backend/src/api/detection.ts`
- Uses `multer({ dest: uploadDir })` where `uploadDir = '../../../uploads'` (relative to that module)
- Middleware sequence:
  1. `upload.single('image')`
  2. `validateFile` (`backend/src/middleware/fileUpload.ts`)
  3. `detectImage` handler (`backend/src/controllers/DetectionController.ts`)

**File:** `backend/src/controllers/DetectionController.ts` (`detectImage`)
- Validates `req.file` exists; otherwise returns `400`.
- Creates:
  - `jobId = uuidv4()`
  - `confidence` from `req.body.confidence` (default `0.5`)
  - `preprocess` from `req.body.preprocess !== 'false'` (default true)
- Calls:
  - `detectionService.processImage(jobId, file.path, { confidenceThreshold, enablePreprocessing })`
- Returns immediately:
  - `res.json({ success: true, jobId, message: 'Image processing started' })`

**Important behavior:** `processImage(...)` is **not awaited**. The HTTP response returns before Python finishes.

### 3.2 Upload request (video)
**Endpoint:** `POST /api/detection/video`

**File:** `backend/src/api/detection.ts`
- Uses `multer` for `video`
- Middleware: `validateFile`
- Handler: `detectVideo`

**File:** `backend/src/controllers/DetectionController.ts` (`detectVideo`)
- Creates `jobId`
- Reads:
  - `confidence` (default `0.5`)
  - `frameInterval` (default `30`)
  - `maxFrames` (default `100`)
- Calls `detectionService.processVideo(jobId, file.path, options)`
- Returns `{ success: true, jobId, message }` immediately

### 3.3 Job status polling
**Endpoint:** `GET /api/detection/status/:jobId`

**File:** `backend/src/controllers/DetectionController.ts` (`getJobStatus`)
- Calls `detectionService.getJobStatus(jobId)`
- Returns `404` if not found, else returns job status JSON

**File:** `backend/src/services/DetectionService.ts`
- Internally stores job state in `private jobs: Map<string, JobEntry>`.
- Status values:
  - `processing`
  - `complete`
  - `error`
- Returns job entry via `getJobStatus()`.

**Important limitation:** Job status exists only in server memory.
- If the backend restarts, existing `jobId`s will be lost.

## 4) Python inference execution (where AI runs)

**File:** `backend/src/services/DetectionService.ts`
- Extends `EventEmitter`.
- Creates:
  - `this.bridge = new PythonBridge()`
  - `this.db = new DatabaseService()`
- `processImage` flow:
  1. Set `jobs[jobId] = { status: 'processing', progress: 0 }`
  2. `const result = await bridge.detectImage(imagePath, options)`
  3. For each `v` in `result.violations`, call `db.saveViolation(v)`
     - Errors while saving violations are caught per-violation and logged.
  4. Set job to `complete` with `progress: 100` and store `result`
  5. Emit `this.emit('complete', { jobId, result })`

- `processVideo` flow is identical but uses `bridge.detectVideo(...)`.

**File:** `backend/src/python-bridge/bridge.ts`
- `detectImage(...)` runs:
  - Script: `<traffic_violation_project>/violation_detector.py`
  - Args:
    - `--image <imagePath>`
    - `--confidence <threshold>`
    - `--preprocess <true/false>`
    - `--json`
- `detectVideo(...)` runs similarly with:
  - `--video <videoPath>`
  - `--interval <frameInterval>`
  - `--max-frames <maxFrames>`

**Critical dependency:**
- The bridge parses **stdout as JSON**:
  - `resolve(JSON.parse(stdout))`
- If Python writes logs/warnings to stdout (instead of stderr), JSON parsing fails.
- If the Python process exits non-zero, the promise rejects with stderr.

## 5) Data persistence (where results are stored)

**File:** `backend/src/services/DatabaseService.ts`

### 5.1 SQLite initialization
- DB path:
  - `process.env.DB_PATH || '../../data/traffic_violations.db'` (resolved relative to this module)
- On construction it calls `this.init()` and creates:
  - table `violations` if not exists
  - indexes on `violation_type`, `plate_text`, `timestamp`

### 5.2 Saving violations
**Method:** `saveViolation(v: any): Promise<number>`

Behavior:
- Waits for DB initialization (`this.ready`).
- Generates `evidenceId = 'EV' + Date.now()`.
- Inserts a row into `violations` with mapping:
  - `timestamp: v.timestamp || new Date().toISOString()`
  - `violation_type: v.type`
  - `plate_text: v.plateText || v.plate_text`
  - `confidence: v.confidence`
  - `detection_confidence: v.detection_confidence || null`
  - `ocr_confidence: v.ocr_confidence || v.plateConfidence || null`
  - `plate_valid: v.plateValid ? 1 : 0`
  - `bbox: JSON.stringify(v.bbox || v.box || [])`
  - `image_path: v.image_path || null`
  - `evidence_path: v.evidence_path || null`
  - `location: v.location || null`
  - `metadata: v.metadata || null`

**Unique constraint risk:**
- Column `evidence_id` is `UNIQUE`.
- Because evidence_id uses `Date.now()` only:
  - If multiple violations insert within the same millisecond, collisions can occur.
  - Collisions result in insert errors (caught at per-violation level in `DetectionService`).

## 6) Analytics + Reports pipeline (how frontend reads data)

### 6.1 Analytics: stats
**Endpoint:** `GET /api/analytics/stats`

**Backend:**
- `backend/src/api/analytics.ts` routes to `AnalyticsController.getStats`
- `backend/src/controllers/AnalyticsController.ts` calls `db.getStats()`

**DB logic:**
- `getStats()` computes:
  - total count
  - `byType` counts
  - `byDate` counts by `SUBSTR(timestamp,1,10)`
  - `totalVehicles` using `COUNT(DISTINCT plate_text) WHERE plate_text IS NOT NULL`
  - compliance:
    - helmetCompliance = based on type `'NO HELMET'`
    - seatbeltCompliance = based on type `'NO SEATBELT'`

### 6.2 Analytics: violations search
**Endpoint:** `GET /api/analytics/violations?type=&plate=&startDate=&endDate=&page=&limit=`

- Implemented in `AnalyticsController.getViolations` → `db.searchViolations(params)`.
- Uses `plate_text LIKE %plate%`.

### 6.3 Reports
**Endpoint:** `POST /api/reports/generate`

**Backend:** `backend/src/controllers/ReportController.ts`
- Reads `startDate`, `endDate`, `types` from request body.
- Creates `reportId` and calls `ReportService.generateReport({ startDate, endDate, types })`.

**ReportService:** `backend/src/services/ReportService.ts`
- Loads:
  - all violations (`db.getAllViolations()`)
  - stats (`db.getStats()`)
- Filters in-memory by date/type.
- Returns report data with:
  - summary + up to 1000 violations

## 7) How frontend displays data (REST)

### 7.1 Detection page (primary flow)
**File:** `frontend/src/pages/Detection.tsx`

Flow:
1. User selects/uploads an image.
2. UI calls:
   - `detectionApi.detectImage(file, confidence, preprocess)`
   - **Endpoint**: `POST /api/detection/image`
3. Response gives `jobId`.
4. UI starts polling every 2 seconds:
   - `GET /api/detection/status/${jobId}`
5. When status is `complete`:
   - reads `status.data.result?.violations`
   - sets `violations` state
6. UI renders:
   - `DetectionCanvas` (bounding boxes)
   - `ViolationList`
   - `DetectionPanel` for details

**Polling error behavior:**
- Some polling callbacks swallow exceptions in `catch { /* retry */ }`.
- If backend throws repeatedly, the UI may keep polling indefinitely.

### 7.2 Analytics page
**File:** `frontend/src/pages/Analytics.tsx`
- Uses React Query with `refetchInterval: 30000`
- Calls `GET /api/analytics/stats`
- Renders charts (BarChart/PieChart/LineChart) and compliance bars.

### 7.3 Reports page
**File:** `frontend/src/pages/Reports.tsx`
- Uses React Query to fetch total stats for display.
- Report generation calls:
  - `POST /api/reports/generate`
- Then displays report summary and downloads JSON.

## 10) Failure points & major issues (what’s most likely to break)

### A) WebSocket is not wired
- Frontend expects `ws://localhost:5001`.
- Backend Express starts only HTTP on `PORT`.
- No bridge from `DetectionService.emit('complete')` to WebSocket broadcast was found in inspected code.

### B) Job status is in-memory only
- `DetectionService.jobs` is a `Map`.
- Any backend restart invalidates all in-progress/finished job IDs.

### C) Python stdout JSON parsing is fragile
- `bridge.ts` requires stdout to be pure JSON.
- Python printing logs/warnings to stdout will fail parsing and mark job as `error`.

### D) Evidence ID collision risk in SQLite
- `evidence_id` is UNIQUE.
- Evidence_id uses only `Date.now()`.
- Under high insertion rate, multiple violations can be saved within the same millisecond.

### E) Polling can run forever on repeated failures
- Frontend polling loops retry on errors without timeout/cancellation.
- If backend never reaches `complete`, the UI may not recover.

### F) Field mapping mismatch between Python output and DB schema
- DB expects certain keys (`v.type`, `v.confidence`, `v.plateText/plate_text`, `v.bbox/box`, etc.).
- If Python returns different field names, data stored may be incorrect or missing.

## 11) “From where to where” data trace (summary)

### Detection image trace (happy path)
1. **User** selects an image in `Detection` page.
2. **Frontend** calls `POST /api/detection/image` (multipart upload).
3. **Express** stores upload into `uploads/` and validates extension/mime/size.
4. **DetectionController** creates `jobId`, triggers `DetectionService.processImage(...)`.
5. **DetectionService** calls **Python** (`violation_detector.py --image ... --json`).
6. **PythonBridge** parses stdout JSON into `DetectionResult`.
7. **DetectionService** iterates `result.violations`:
   - calls `DatabaseService.saveViolation(v)` for each violation.
8. **SQLite** persists rows into `violations` table.
9. **DetectionService** marks job `complete` and stores `result` in memory.
10. **Frontend** polls `GET /api/detection/status/:jobId` until `complete`.
11. **Frontend** renders the bounding boxes + violation list.

### “Broadcast to frontend” (current reality)
- Real-time broadcast via WebSocket likely **does not occur**.
- The detection UI currently relies on **polling**.
